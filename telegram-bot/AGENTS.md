## Project invariants (Sera FX bot ŌĆö do not break)

1. **Sign verbatim.** EIP-712 payloads from Sera (`route_params` from `/swap/quote`,
   the `/orders/preview` response) are signed exactly as returned. Never reconstruct
   or "fix" them client-side. **Sole documented exception:** VL batch legs ŌĆö the
   preview only accepts standalone uuid_int encodings, so `liquidity-service.ts`
   swaps ONLY the `uuid` field to the VL encoding before signing (verified live
   2026-07-10); everything else stays verbatim.
2. **Address casing.** Read endpoints (`/balances` etc.) take `owner_address` in
   **lowercase**; signed payloads use the checksummed address as-is.
3. **Money moves through one path only.** Every fund-moving flow creates a
   `pending_actions` row (single-use, TTL) ŌåÆ confirmation card ŌåÆ user's button tap ŌåÆ
   `bot/callbacks.ts` ŌåÆ `PrivySigner`. Mastra tools are read-only or prepare-only;
   they must never sign or submit.
4. **Identity from requestContext.** Mastra tools resolve the user from
   `requestContext` values set by the bot layer ŌĆö never from model-provided arguments.
5. **No hardcoded chain data.** Chain id, contract addresses, and the EIP-712 domain
   come from `GET /config` per network.
6. **Sera is spot-only.** No leverage, margin, or stop-loss anywhere in code or copy.
   - **Private keys never persist.** `/exportkey` `/importkey` use Privy's
     high-level `wallets().exportPrivateKey()` / `wallets().import()` (HPKE is
     handled inside the SDK), so the key is only ever plaintext in one Telegram
     message: the export message auto-deletes (60 s), the import message is
     deleted immediately. Never write the key to the DB. Export/import require the
     wallet to have a **P-256 owner** (`WALLET_AUTH_PRIVATE_KEY`/`_PUBLIC_KEY`,
     generated by `scripts/gen-wallet-auth-key.ts`); the feature disables itself
     cleanly when those are unset. Setting an owner does **not** break app-secret
     `signTypedData` (verified live), so trading is unaffected. Key-op *failures*
     are safe to log/show via `describeKeyOpError` (a failed export/import carries
     no plaintext key ŌĆö it's only in a successful, SDK-decrypted response, or in
     local memory never passed to Privy); use that, don't log the raw plaintext key.
7. **Precision.** Validate against `tick_precision`/`quantity_precision` before any
   API call; `rounding_mode` is `reject_extra_precision` ŌĆö never silently round.
8. **Order retries reuse the same `order_id`** (server dedupes); `uuid_int` must be
   `encodeUuidInt(order_id)` (`src/sera/uuid-int.ts`) or the API rejects it.
